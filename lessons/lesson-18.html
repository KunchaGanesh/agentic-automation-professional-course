Process Variables & User Task Configuration in Maestro

Mastering Data Flow and Human Interaction in UiPath Maestro

  • Understanding how process variables and user task configuration work in UiPath
  • Maestro is essential for building workflows that are robust, efficient, and adaptable.
  • These foundational concepts ensure that data flows reliably between automated and human-driven steps, enabling seamless collaboration and precise control across complex business processes.

By mastering these elements, you’ll be able to orchestrate processes that are both flexible and resilient—capable of handling dynamic data and real-time human input with ease.

This lesson will guide you through essential concepts and tools for managing variables and configuring user tasks in Maestro. You will gain hands-on experience and test your understanding with practical examples and knowledge checks.

What you’ll learn

  1. Distinguish between global, local, and task-specific variables in Maestro.
  2. Use the Variables and Expression Editor for managing data.
  3. Configure user tasks with dynamic assignment and data binding.
  4. Apply concepts through practical workflow examples.
  5. Test your understanding with real-world knowledge checks.

Understanding Variables in Maestro

Variables in Maestro are declared data holders that persist information throughout the execution of a process. They serve as the primary mechanism for passing data between tasks, making decisions in gateways, and feeding dynamic values into integrations.

By carefully managing variables, you ensure that the right information is available at every step of your workflow, supporting both automation and human decisionmaking.

Maestro Variables: Core Concepts

Review these essential concepts to understand how variables function within Maestro and how they are managed.

What is a variable in Maestro?

A declared data holder that persists information throughout process execution.

Where are variables created and managed?

Through the Variables and Expression Editor.

What types of data can Maestro variables store?

Typed values, JSON structures, and expressions.

Variable Scopes: Where and How Data Lives

Variables in Maestro can exist at different scopes, each serving a unique purpose in process design. Expand each section below to learn how global, local, and taskspecific variables operate and when to use them.

Global (Process-Level) Variables

Global variables are declared at the process level and are accessible from every task, gateway, subprocess, and integration within that process. They are created once and can be read or updated by any component of the workflow throughout its entire execution lifecycle. When to use global variables: When the same piece of data is needed by more than one task, gateway, or integration When a value drives a routing decision at a gateway — the gateway must be able to read it When a value needs to be carried from the beginning of the process to the very end — for example, from the intake form all the way to the final notification email

When the value is part of the process identity — such as a claim number, order ID, or requester email that defines what the process instance is about

How they work: Global variables are defined in the Variables panel at the process level. They are named, typed, and optionally given a default value. The Variables and Expression Editor is used to read and write them from within any task or gateway in the process.

Examples:

Amount — stores the monetary value of a purchase order, read by the routing gateway and referenced in the approval notification RequesterEmail — stores the submitter's email address, used at the end of the process to send a decision notification ApprovalStatus — updated by the approver's User task and read by the downstream gateway to route to the approved or rejected path ClaimID — the unique identifier for an insurance claim, referenced by every task in the process

Common mistake to avoid: Declaring every variable as global by default. Global scope should be reserved for data that is genuinely shared. Overusing global variables creates a cluttered process that is harder to maintain and debug.

Local / Subprocess-Level Variables

Local variables exist within the boundaries of a subprocess. A subprocess is a self-contained logical block within a larger process — it encapsulates a set of tasks that perform a specific function, such as document verification, fraud detection, or payment processing. Variables declared inside a subprocess are not automatically visible to the parent process. When to use local variables: When a variable holds data that is only relevant within a specific subprocess and has no meaning outside it When you want to encapsulate complexity — keeping internal calculations, intermediate results, and temporary flags hidden from the parent process When building reusable subprocesses that can be dropped into different parent processes without carrying unwanted variable dependencies

How outputs are shared with the parent process: When a subprocess needs to pass a result back to the parent, two mechanisms are available: 1. Output Variables — explicitly declared as outputs of the subprocess. The parent process maps these to its own variables when the subprocess completes. 2. End Event Variables — automatically propagated to the parent process when the subprocess reaches its end event, without requiring explicit mapping.

Examples:

IsDocumentValid — a boolean variable inside a document verification subprocess. True if all required documents are present and valid, False otherwise. Declared as an output variable so the parent gateway can route based on the result. VerificationErrorCode — an internal variable that holds the specific reason a document failed verification. Used only within the subprocess for logging. Not exposed as an output because the parent only needs the pass/fail result. FraudScore — calculated inside a fraud detection subprocess and exposed as an output variable so the parent process can route to manual review if the score exceeds a threshold.

Common mistake to avoid: Assuming subprocess variables are automatically available in the parent process. Unless explicitly declared as output variables or end event variables, subprocess data stays within the subprocess scope and is not accessible outside it.

Task-Specific Variables

Task-specific variables are defined within an individual task — such as a User task, RPA task, or API call — and are local to that step. They hold data that is produced by the task and is only needed during that step's execution. They do not persist beyond the task unless explicitly written into a global variable. When to use task-specific variables: When a task produces a value that is only needed within that task and has no relevance elsewhere in the process For temporary results: error messages, intermediate calculations, validation flags, or API response fields that are used to determine a task outcome but not needed downstream When capturing form outputs or API responses before deciding which fields to promote to global variables

How they are created: Inside the task configuration panel, use Add new to define a new variable local to that task. Use Set variable value to update an existing global variable with a value derived within the task. The Variable or Expression Editor supports binding values from JSON structures, Data Fabric fields, and API responses.

Examples:

ValidationErrorMessage — holds a temporary error message generated if a form field fails validation. Displayed to the user within the task. Not needed after the task resolves the error. APIResponseCode — stores the HTTP response code from an external API call made within the task. Used to determine whether to retry or proceed. Not relevant to any other task in the process. CalculatedTax — an intermediate calculation performed inside a financial processing task. The final result is written into the global variable TotalAmount. The intermediate value itself is task-specific. DocumentUploadStatus — a flag set within a document upload task indicating whether the upload succeeded or failed. Used within the task to trigger a retry. Not needed downstream once the task completes successfully.

Common mistake to avoid: Using task-specific variables to store data that is actually needed downstream. If a value produced in Task 3 is needed in Task 7, it must be written into a global variable — not kept as a task-specific variable, which will be out of scope by the time Task 7 executes.

Configuring User Tasks in Maestro

User tasks in Maestro represent steps where a human must take action—such as reviewing, approving, or providing information. These tasks pause the automated process until the assigned person completes their action, ensuring that human judgment and input are seamlessly integrated into the workflow.

Proper configuration of user tasks is crucial for ensuring that the right person receives the right work at the right time, and that all necessary data is captured and routed back into the process.

Key Aspects of User Task Configuration in Maestro

User tasks in Maestro have several important configuration options that determine how they function within a workflow. Expand each section to learn more about implementation type, dynamic assignees, and data binding.

Implementation Type

When adding a User task to a Maestro process, the key configuration is Implementation → Type: Create action app task. This connects the task to an Action Center Action app — the interface that the assigned user will see and interact with.

Dynamic Assignees

Rather than hardcoding a specific person or group, Maestro supports data-driven assignment so that the correct person is automatically determined at runtime based on process data.

Assignment is handled on the Action Center side using the Assign Tasks activity. Assignment criteria options: Single User: Assign to a specific individual by username or email. All users in a group: The task appears in everyone's queue within a group. Workload based: Assigned to the group member with the lightest current workload. Round-Robin based: Assigned to the next person in a rotating sequence.

The assignee information is passed via variables — for example, the process variable manager_email is bound to the task's assignment configuration. The same assignment options exist both in the Activities panel and the Action Center user guide.

Task Inputs and Outputs

User tasks have two-way data binding:

Inputs: Data from process variables that is pre-populated into the Action app form so the reviewer has full context. Outputs: Data captured from the user's interaction — decisions, comments, corrections — that is written back into process variables. Use the Variable or Expression editor to bind each field.
