API Workflows: Design, Build & Secure Integration

Unlocking Cloud-Native Integration: The

Power of API Workflows in UiPath

API workflows are transforming how organizations connect systems, automate business logic, and enable real-time collaboration between applications. Unlike traditional automation that relies on attended or unattended robots, API workflows operate entirely in the cloud, providing a scalable, governed, and reusable way to expose business logic as APIs. This approach eliminates the need for dedicated robots or servers, making automation more flexible, responsive, and easy to manage. By bridging UiPath with external systems and AI agents, API workflows unlock new possibilities for seamless, cloud-native integration.

API Workflow Architecture & Key Benefits

An API workflow in UiPath is a cloud-hosted, serverless workflow that exposes business logic as a secure, governed, and reusable API. These workflows run on UiPath Automation Cloud infrastructure, meaning there are no robots or virtual machines to manage. All input and output is handled through structured JSON schemas, creating a clear contract between the API and its callers. API workflows execute synchronously, so the calling system receives results in real time. Every execution is logged and managed centrally in Orchestrator, ensuring governance and auditability. API workflows are also composable—they can call other APIs, invoke RPA processes, and interact with UiPath’s Integration Service and Data Fabric, making them a powerful tool for modern automation.

Key Features and Use Cases of API Workflows

API workflows in UiPath Automation Cloud offer a modern, scalable, and governed approach to integrating automation with external systems. Explore the main features and typical scenarios where API workflows deliver value.

Serverless & Scalable

API workflows run entirely on UiPath Automation Cloud's serverless infrastructure. There is no need to provision robots or manage virtual machines. The platform automatically scales to handle demand, making it easy to support both small and large integration workloads. This serverless approach reduces operational overhead and allows teams to focus on building business logic rather than managing infrastructure.

JSON-Based Interface

All input and output for API workflows is defined using JSON schemas. This creates a welltyped contract between the API and its callers, ensuring that data is structured, validated, and easy to document. Using JSON schemas also enables automatic documentation and helps prevent integration errors by making data requirements explicit.

Governed & Auditable

API workflows are centrally managed and monitored in Orchestrator. Every execution is logged, and IT administrators can control access, monitor usage, and audit activity from a single location. This governance ensures that integrations are secure, compliant, and easy to manage across the organization.

Common Use Cases

API workflows are used in a variety of real-world scenarios, such as:

Order processing: Receive an order payload, validate, write to ERP, and return confirmation.

Customer data sync: Accept a customer record, check for duplicates, merge or create, and return a canonical ID. Document processing: Receive a document path, extract structured data using AI, and return key-value pairs. ML inference: Call an AI model with input features and return prediction scores. Real-time notifications: Trigger emails or alerts and return delivery status.

Designing Input & Output Schemas for API Workflows

Schema design is the foundation of every API workflow. Well-typed, explicit input and output schemas ensure that any caller—whether an AI agent, external application, or another UiPath process—knows exactly what data to send and what to expect in return. Clear schemas prevent errors, enable automatic documentation, and make your APIs easier to use and maintain. By defining robust JSON contracts, you create reliable, reusable integrations that stand the test of time.

Schema Design Best Practices and Examples

Learn how to define, structure, and manage schemas in Studio Web to ensure robust and maintainable API workflows. Explore practical examples and key principles for effective schema design.

In Studio Web, you define input and output schemas using the Data Manager panel. You can generate a schema automatically by uploading a sample JSON payload, which infers field names, data types, and nesting. Alternatively, you can add each property manually, specifying its name, data type (string, number, boolean, array, object), whether it is required, and any nested structures. This flexibility allows you to quickly create accurate schemas that match your integration requirements, ensuring that your API workflow is both robust and easy to consume.

For an order processing API, a typical input schema might look like this: { order_id: string (required), customer_email: string (required), items: [ { sku: string, quantity: number, unit_price: number } ], shipping_address: { street: string, city: string, postal_code: string } }. The output schema could be: { status: string (enum: 'confirmed' | 'pending' | 'error'), order_id: string, estimated_delivery: string (ISO date), error_message: string (nullable) }. These examples show how to structure data for clarity and validation.

Be explicit about required fields—mark them as required if the workflow cannot execute without them. Use enums for fields with a fixed set of valid values, such as status. Nest related data into objects or arrays to keep schemas organized and logical. When making breaking changes, always version your schemas by creating a new API workflow version rather than modifying the existing one. This prevents disruptions for existing callers and ensures backward compatibility.

Building & Testing API Workflows in Studio Web

Once your schemas are defined, you can build and test API workflows directly in Studio Web. The process involves creating a new API workflow, implementing business logic with activities, mapping inputs and outputs, and validating your

workflow with sample data. This hands-on approach ensures your workflow is robust and ready for deployment before it goes live.

Introduction to Creating an API Workflow

This lesson will guide you through the step-by-step process of creating an API workflow in Studio Web, from initial setup to publishing your integration.

  1. 1
    Start in Studio Web
    Open Studio Web and select Create New → API Workflow to begin building your integration.
  2. 2
    Define Input Schema
    In the Data Manager panel, configure the input schema using either the Generate From Payload feature or by manually adding properties.
  3. 3
    Define Output Schema
    Set up the output schema in Data Manager, ensuring all expected response fields are included and well-typed.
  4. 4
    Add Business Logic Activities
    On the workflow canvas, add activities such as HTTP Request, Data Manipulation, Connector, or Script to implement your business logic.
  5. 5
    Map Inputs and Outputs
    Use the Expression editor to map input fields to activity parameters and activity outputs to the output schema, ensuring data flows correctly through the workflow.
  6. 6
    Test with Sample Data
    Before publishing, test your workflow with sample JSON input directly in Studio Web to validate its behavior and catch any errors early.

Review and Publish

After successful testing, review your workflow for completeness and publish it to Orchestrator for use in production integrations.

Key Development Tools for API Workflows

Explore the essential tools that help you build, customize, and maintain effective API workflows. Each tool serves a unique purpose, from handling data and logic to managing errors and custom code.

Activities

Activities are the building blocks of your workflow. Use HTTP requests, data transformation, connector integrations, and conditional logic to implement business logic and connect to external systems. Each activity can be configured to handle specific tasks, making your workflow modular and easy to maintain.

Expression Editor

The Expression editor allows you to write expressions to map data, transform strings, calculate values, and handle conditional logic inline. This makes it easy to customize how data flows through your workflow. Using expressions ensures that your workflow can adapt to different input scenarios and produce the correct output every time.

JavaScript Script Activity

For complex data transformations that are difficult to achieve with standard activities, use the JavaScript Script activity. This lets you write custom code directly within your workflow. JavaScript scripts provide flexibility for advanced scenarios, such as manipulating nested JSON or performing calculations.

Try/Catch

Wrap critical activities in Try/Catch blocks to handle errors gracefully. This ensures that your workflow produces meaningful error responses instead of failing silently. Good error handling improves reliability and makes your API workflow more robust in production environments.

Consuming API Workflows: From Orchestrator & Agents

After building and publishing your API workflow, it can be consumed in multiple ways. You can trigger workflows directly from Orchestrator, providing input data and viewing execution logs. Alternatively, AI agents can use API workflows as tools, calling them at runtime based on user intent and integrating results into their responses. Understanding these consumption patterns helps you design workflows that fit seamlessly into broader automation strategies.

Deployment, Security & Network Configuration

Securing your API workflows is essential for safe and reliable integration. When deploying API workflows, you must consider network security, especially when communicating with external systems. IP allowlisting is required for outbound connections to systems outside UiPath, while internal UiPath communications are pre-secured. Understanding when and how to configure allowlisting ensures your integrations remain both accessible and protected.

Key Security and Network Considerations for API Workflows

Understanding the security and network requirements for API workflows in UiPath is essential for maintaining a secure and functional automation environment. The following sections outline the main considerations regarding internal and external communication, allowlisting requirements, and activity types with their respective authentication and network paths.

Internal vs External Communication

Internal UiPath calls—such as communication between API workflows and Orchestrator—are platform-managed and do not require allowlisting. However, when API workflows connect to external systems, allowlisting is necessary to permit traffic from UiPath’s serverless infrastructure. This distinction helps maintain security while enabling flexible integration with both internal and external endpoints.

Allowlisting Requirements

Allowlisting is required when API workflows use HTTP Request or Connector activities to communicate with systems outside UiPath. For HTTP Request activities using manual authentication, you must add UiPath’s serverless static IPs to the external system’s allowlist. For connector-based activities, use Integration Service IPs. Proper allowlisting ensures that only authorized traffic can reach your external APIs, reducing security risks.

Activity Types & Authentication

Here is a summary of activity types, authentication methods, outbound paths, and when allowlisting is needed:

HTTP Request (Manual Auth): Outbound via Serverless robots – Allowlisting needed HTTP Request (Connector-based): Outbound via Integration Service – Allowlisting needed Connector activity: Outbound via Integration Service – Allowlisting needed Internal UiPath calls: Platform-managed, within platform – No allowlisting needed
