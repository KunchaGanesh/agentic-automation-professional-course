Securing Agent Operations: Environment Variables and Asset Management

Securing Agent Operations: The Foundation of Trustworthy Enterprise Automation

UiPath coded agents establish a secure connection to UiPath as a core step in the deployment process. Developers can use UiPath assets for secure storage and management of sensitive data. Assets in Orchestrator are encrypted at rest by default, and all asset types of Credential and Secret are encrypted using the AES 256 algorithm.

You can configure your project with environment variables using secret assets, eliminating the need for direct code modifications. Navigate to your agent's process in Orchestrator to define and manage API keys, runtime parameters, and other configuration values required by your agent's code. Using the env. prefix when passing credentials in the CLI keeps secret values out of shell history and process listings.

Orchestrator also supports integration with external credential stores such as CyberArk. At runtime, Orchestrator securely retrieves the key from CyberArk via the asset and makes it available to your agent as an environment variable.

This lesson will guide you through essential practices for configuring and securing coded agents in automation environments.

Configuring Environment Variables for Your Coded Agents

Environment variables are a best practice for managing configuration and secrets in coded agent projects. Instead of embedding sensitive values directly in your code, you define environment variables in Orchestrator, which securely injects them into your agent’s runtime environment. This approach allows you to manage API keys, runtime

parameters, and other configuration values centrally, making updates seamless and reducing security risks. By leveraging environment variables, you ensure that your agents are flexible, portable, and compliant with enterprise security standards.

Naming Conventions

Use environment variables for easy configuration changes without modifying code, improving adaptability and security. For cross-platform workflows, only rely on environment variables passed to the process or set during execution — do not depend on user-level variables as they are not available on Linux or macOS. Test that all required variables are available at process startup and document any environment prerequisites clearly.

Scope and Access Control

  • On Windows, the Get Environment Variable activity retrieves variables from User Environment
  • Variables with fallback to Process Environment Variables.
  • On cross-platform projects, it retrieves Process Environment Variables only. For portable workflows, design assuming Process-level scope only to ensure consistent behaviour across platforms.

How to Securely Use Assets in Coded Agent Workflows

This lesson will guide you through the process of securely managing and using assets— such as secrets, credentials, and parameters—in your coded agent workflows to ensure security and maintainability.

  1. 1
    Define Required Assets
    UiPath assets support five types: Text (stores strings), Bool (true or false values), Integer (whole numbers), Credential (usernames and passwords), and Secret (API keys — usable only with coded agents). Identify which type suits each value your agent requires before creating them in Orchestrator.
  2. 2
    Create and Manage Assets
    In Orchestrator, navigate to the Assets page and click Add to create a new asset. Select the asset type, enter the value, and assign it to the correct folder. Once an asset is created, you cannot change its asset type. Only the asset value and name can be edited after creation.
  3. 3
    Reference Assets in Code
    UiPath SDKs provide developers with the necessary logic to invoke and interact with UiPath resources directly within their code. Developers can use UiPath assets for secure storage and management of sensitive data. Asset names must match exactly — including capitalisation — as stored in the Orchestrator database for the asset to be retrieved correctly.
  4. 4
    Retrieve Asset Values at Runtime
    At runtime, Orchestrator securely retrieves the asset value and makes it available to your agent. For credential assets, the activity uses the same authentication context as the robot under which it runs. The robot must have the required permissions for the asset to be retrieved successfully.
  5. 5
    Audit and Monitor Asset Usage
    All changes to assets are audited and can be viewed on the Audit page in Orchestrator. All tool usage by agents is tracked and recorded in a consistent manner — by reviewing execution logs, you can detect anomalies such as an agent calling an endpoint it normally does not.

Managing API Keys and Runtime Parameters at Scale

You can define environment variables that allow you to make configuration changes to your agent without modifying the code directly. To specify environment variables for coded agents, use the Environment Configuration text field in the process settings in Orchestrator. You can use this field to reference secrets such as API keys by entering the value directly or by referencing a secret-type asset located in the same folder as the agent.

When environment variables are passed as raw strings to a process or job, only the variable key is returned — this prevents sensitive values from being exposed. You can enhance security further by ensuring that secret key names end in specific strings such as SECRET or TOKEN, which causes asterisks to be displayed instead of the secret value when the process is subsequently edited.

Runtime arguments can be provided at the process level, job level, or trigger level. Arguments are validated against the provided JSON schema, and you can toggle

between viewing arguments and the underlying JSON schema directly in Orchestrator.

Securely Managing and Updating an API Key for a Coded Agent Learn how to securely store, reference, update, monitor, and audit API keys for coded agents to minimize risk and ensure compliance.

  1. 1
    Store API Key as Asset
    The Secret asset type stores secrets such as API keys and can be used only in conjunction with coded agents. Store the API key in an asset named clearly — for example MY_API_KEY. Assets in Orchestrator are encrypted at rest by default, and the UI shows only a masked placeholder, so the secret is never stored in plain text.
  2. 2
    Reference Asset in Configuration
    In the Environment Configuration field of your agent's process, reference the secrettype asset located in the same folder as the agent. Typing =% after the secret key displays an ASSETS/ autocompletion suggestion, allowing you to select the desired asset from a drop-down list.
  3. 3
    Update Key in Orchestrator
    To update the secret value, delete the asterisks in the Environment Configuration field and enter the new string. You can define environment variables that allow you to make configuration changes to your agent without modifying the code directly — no redeployment is needed.
  4. 4
    Monitor Agent Usage
    Once deployed, you can schedule, trigger, and monitor agent execution in Orchestrator. All tool usage by agents is tracked and recorded in a consistent manner. By reviewing execution logs, you can detect anomalies such as an agent calling an endpoint it normally does not.
  5. 5
    Audit Access and Usage
    The Audit page in Orchestrator displays the audit trail for actions performed by all entities, filterable by Component, User, Action, and Time. All changes to assets are audited and can be viewed on the Audit page. Audit log data is stored in Orchestrator for at least six months and is available for a total period of 24 months, with monthly archives available for download.
